# 问题及解决措施

## 2026-05-01

- 问题：CE 候选剪枝按 token 前缀长度区分 template 与 search，UWB token 如果追加在 search 后面会被 CE 切分逻辑误判。
- 处理：在 `VisionTransformerCEUWB` 中将 UWB token 放在 template 与 search 之间，同时扩展 `global_index_t`，确保 UWB token 始终位于 template 侧。

- 问题：动态剪枝如果逐样本保留 token 数不同，会破坏 batch 张量形状。
- 处理：初版采用 batch 内统一保留率，保留率由 batch 平均 `uwb_conf_pred` 动态计算。

- 问题：测试时需要看到剪枝后的 search 图。
- 处理：剪枝模块输出 `uwb_layer0_removed_indexes_s`，测试 tracker 在 `--debug 1` 下注册 `uwb_pruned_search_layer0` 到 visdom。

- 问题：本机未启动 visdom 时，`--debug 1` 会导致 `self.visdom` 为 `None` 但仍进入 visdom 注册分支。
- 处理：UGTrack 测试 tracker 在 visdom 初始化失败时自动降级到本地 debug 图片输出；启动 visdom 后仍走 visdom 可视化分支。

- 问题：早期实现把非 CE UWB token 和剪枝逻辑接入了 `lib/models/ostrack/base_backbone.py`，不符合 AGENTS.md 第 8 条。
- 处理：已还原 `lib/models/ostrack/` 下改动，并在 `lib/models/ugtrack/` 下新增 `vit_uwb.py`、`ostrack_uwb.py`、`uwb_pruning.py` 承载 UGTrack 专用逻辑。

- 问题：visdom 未启动时，`debug/` 目录原先只保存整帧跟踪结果图，不能查看剪枝前后 search 对比。
- 处理：新增本地 debug 保存逻辑，分别输出 `debug/search_before_pruning/*.jpg` 和 `debug/search_after_layer0_pruning/*.jpg`。
