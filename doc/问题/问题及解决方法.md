- 每次执行 `doc/进度/进度.md` 中的步骤时，如果遇到问题，必须在本文新增或更新记录。

### 步骤 8：权重加载模块

- 状态：已解决
- 问题现象：UGTrack Stage-2 训练里同时存在三类权重加载入口：当前实验断点续训、阶段一 UWB checkpoint 初始化、OSTrack 视觉分支初始化。当前代码执行顺序和命名容易让实验起点不清晰，特别是当前实验目录已有 checkpoint 时，会先恢复当前实验，再加载前序 checkpoint，可能覆盖或混淆预期初始化来源。
- 原因判断：`lib/train/train_script_ugtrack.py` 固定以 `load_latest=True` 启动 trainer；同一次 trainer 启动中又根据 `settings.project_path_prv` 设置 `load_previous_ckpt=True`。模型构建阶段还会根据 `MODEL.PRETRAIN_FILE` 先加载 MAE 或 OSTrack 权重。因此“resume”和“init”的语义没有明确隔离。
- 解决方法：目标训练场景已明确：Stage-1 不加载预训练模型，UWB 分支随机初始化后训练出最终权重；Stage-2 只支持两类初始化实验：1. UWB 加载 Stage-1 最终权重，OSTrack 视觉分支加载 MAE ViT 权重且 head 随机初始化；2. UWB 加载 Stage-1 最终权重，OSTrack 视觉分支加载已训练 OSTrack checkpoint，包含 ViT 与 head 权重。断点续训保留为训练恢复机制，但与初始化实验互斥。已修改 `lib/train/trainers/base_trainer.py`：当前实验 checkpoint 成功 resume 后，跳过 `project_path_prv` 前序权重加载；只有没有 resume 到当前实验 checkpoint 时，才加载 Stage-1 UWB 权重。
- 验证结果：`script/check_ugtrack_weight_loading.py` 已验证三类权重初始化路径；`base_trainer.py` 已通过语法检查。Stage-2 smoke 已完成两轮验证：第一次无当前实验 checkpoint 时加载 Stage-1 UWB 权重并完成 1 epoch；第二次同一输出目录已有当前实验 checkpoint 时，日志显示 `Skip previous checkpoint loading because current experiment checkpoint was resumed.`。
- 相关文件：`tracking/train.py`，`lib/train/run_training.py`，`lib/train/train_script_ugtrack.py`，`lib/train/trainers/base_trainer.py`，`lib/models/ugtrack/ugtrack.py`，`lib/models/ostrack/vit.py`，`lib/test/tracker/ugtrack.py`，`experiments/ugtrack/vitb_256_mae_ep300_ostrack_uwb_conv1d_token.yaml`，`doc/方法/加载指定权重.md`
- 记录时间：2026-04-24

### 步骤 8：权重加载验证脚本忽略 BatchNorm 计数 buffer

- 状态：已解决
- 问题现象：运行 `python -X utf8 script/check_ugtrack_weight_loading.py` 时，Stage-1、Stage-2 MAE、Stage-2 OSTrack 的主要加载流程均通过，但最终比较 OSTrack tracker checkpoint 时在 `box_head.*.num_batches_tracked` 上报 tensor differs。
- 原因判断：`num_batches_tracked` 是 BatchNorm 的训练步计数 buffer，不是模型权重，也不是 running mean / running var 这类归一化统计量。它可能因模块初始化或加载路径表现不同，不适合作为“OSTrack tracker 权重是否加载成功”的严格等价校验项。
- 解决方法：修改 `script/check_ugtrack_weight_loading.py`，在 `_assert_equal_tensor_dict()` 中跳过所有以 `num_batches_tracked` 结尾的 key；其余参数和 buffer 仍保持严格 `torch.equal` 校验。
- 验证结果：重新运行 `python -X utf8 script/check_ugtrack_weight_loading.py` 后，Stage-1、Stage-2 MAE、Stage-2 OSTrack 三类权重加载验证均通过。
- 相关文件：`script/check_ugtrack_weight_loading.py`
- 记录时间：2026-04-24

### 步骤 8：训练入口显式控制 resume

- 状态：已撤回
- 问题现象：新实验和断点续训都走 `load_latest=True`，用户无法在命令行显式禁止从当前实验目录恢复 checkpoint；另外 `tracking/train.py` 会把未传的 `--script_prv` 格式化为字符串 `None` 继续传给 `run_training.py`，存在误设置 `project_path_prv=train/None/baseline` 的风险。
- 原因判断：训练入口没有 `resume` 显式开关，且命令字符串无条件拼接可选参数。
- 解决方法：曾尝试新增 `--resume 0/1` 参数；用户确认不希望增加启动参数，因此已撤回该改动。当前保留默认 `load_latest=True`，依靠“新实验使用新 config 或清空当前实验 checkpoint 目录”来避免误断点续训。
- 验证结果：撤回后相关 Python 文件已通过语法检查。
- 相关文件：`tracking/train.py`，`lib/train/run_training.py`，`lib/train/train_script_ugtrack.py`
- 记录时间：2026-04-24

### 步骤 8：Smoke 测试 Python 环境缺少 PyYAML

- 状态：已解决
- 问题现象：运行 `python -X utf8 script/check_ugtrack_stage1_train_smoke.py --config vitb_256_mae_ep300_ostrack_uwb_conv1d_token ...` 时，脚本在 `import yaml` 处失败，报 `ModuleNotFoundError: No module named 'yaml'`。
- 原因判断：当前命令执行环境使用的 Python 不是已配置完整依赖的 `ostrack` 环境，缺少 PyYAML。该失败发生在脚本启动阶段，尚未进入训练或权重加载逻辑。
- 解决方法：改用项目训练环境运行，例如在终端中先 `conda activate ostrack`，或使用 `conda run -n ostrack python ...` 重新运行同一 smoke 命令。
- 验证结果：使用 `conda run -n ostrack python -X utf8 script/check_ugtrack_stage1_train_smoke.py ...` 后，脚本进入训练流程并完成 1 epoch smoke。
- 相关文件：`script/check_ugtrack_stage1_train_smoke.py`
- 记录时间：2026-04-24

### 步骤 8：Smoke 退出时 wandb 临时目录权限警告

- 验证结果：待确认是否需要进一步处理；本次 smoke 主流程已完成。
